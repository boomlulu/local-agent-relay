## 已知边界

### 1. 密集中文 OCR —— 不行，且会「自信地编造」

- **案例（2026-06-07）**：让 Gemma「识别图里所有文字并按游戏玩法排版」。原图是桌游《卡卡利科·卡卡夜未眠》角色技能速查表（7-15 人狼人杀类，约 40 角色，分镇民/外来者/爪牙/恶魔 + 说书人）。
- **结果**：读成《白羊神话》，编出「星座 / 白羊座 / 星灵 / 战魂」等图里**根本不存在**的内容；约 40 个真实角色一个没认出。
- **评分（对「识别所有文字」目标）**：总体 ≈ **2/10**。其中识别准度 0.5/10、排版 8.5/10、稳定性（修复后）OK。
- **根因**：4bit 量化 + 通用 VLM 的视觉编码器分辨不了密集小号中文。**它是「看不清」，不是「不会说」。**
- **危险点**：输出排版精美、措辞自信 → 比明显失败更坑（看起来像真的，会误导人）。
- **怎么办**：要真 OCR，用 Apple Vision / PaddleOCR 取字，再（可选）交 LLM 排版。**提示词救不了**（见下）。

### 2. 复读退化 (repetition loop)

- **案例（2026-06-07，同上任务）**：未加抗复读参数时，输出「白骨塔中」复读约 500 遍直到 token 耗尽。
- **修复**：mlx_vlm 调用加 `--repetition-penalty 1.3 --repetition-context-size 64` + 提高 `max_tokens`（默认 1024）。已设为默认，env 可调。
- **效果**：死循环消失（任何中文片段重复 ≤ 3 次），输出完整正常结束。**但识别准度不变**——抗复读只治「卡死」，不治「读不准」。

## 提示词能做到 / 做不到（重要认知）

**提示词改的是「行为」，加不了「感知」。**

- **能**：减少编造（"只照抄图中真实存在的文字，不确定标 `[不清]`，禁止脑补"）、逼它逐行抄、让它承认读不清、改善排版。乐观把 2 分拉到 3-4 分，清晰大字图能更高。
- **不能**：让它读出「看不清的字」。视觉编码器分辨不了的内容，再好的提示词也变不出来。
- **结论**：**靠提示词拿不到满分。** 满分必须换感知层（真 OCR）。识别交 OCR、排版交 LLM，是这类任务的上限路径。

## Gemma 执行器调参旋钮（环境变量）

| 变量 | 默认 | 作用 |
|---|---|---|
| `LOCAL_AGENT_RELAY_GEMMA_REP_PENALTY` | `1.3` | 抗复读惩罚（>1 生效，越大越压重复） |
| `LOCAL_AGENT_RELAY_GEMMA_REP_CTX` | `64` | 抗复读回看 token 数 |
| `LOCAL_AGENT_RELAY_GEMMA_TEMP` | `0.2` | 采样温度 |
| `LOCAL_AGENT_RELAY_GEMMA_MAX_TOKENS` | `1024` | gemma 执行器输出上限 |
| `LOCAL_AGENT_RELAY_GEMMA_TIMEOUT` | `240` | 子进程超时（秒） |
